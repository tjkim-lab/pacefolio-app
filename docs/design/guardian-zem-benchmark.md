# 보호자 온보딩 — ZEM 벤치마크 분석 & PACEFOLIO 변환

> 작성 2026-07-19 · 대상: 보호자(학부모) 모바일 온보딩·가입·홈
> 목적: ZEM(젬) 앱의 **디자인 품질·UX 흐름**을 벤치마크하여 PACEFOLIO 보호자 화면을 개선.
> ⚠️ ZEM은 **참고만** — 에셋·카피·색·일러스트·개인정보는 복제하지 않는다. PACEFOLIO 고유 시각 언어(움직임·경험·도전·기술 성장·뱃지·코치의 관찰·가족의 응원)로 변환한다.

## 0. 한 줄 요약

ZEM에서 **가져올 것** = 넓은 여백 · 한 화면 한 행동 · 2줄 제목/2~3줄 설명 · 하단 고정 단일 CTA · 권한 요청 전 설명 · 필수/선택 동의 분리 · 부드러운 카드(큰 라운드·약한 그림자).
ZEM에서 **버릴 것** = **아이 연결(검색·연결코드(아이용)·QR·아이 승인·타 보호자 승인) 전부**.

> ⚠️ **모델 개정 (2026-07-19)**: 초기엔 "학원 선등록 원생을 전화번호로 자동조회"였으나 유저 확정으로 변경 —
> **기존 원생 데이터를 매칭하지 않는다. 학부모가 초대코드로 학원에 진입 → 휴대폰 본인인증 → 아이를 직접 등록**한다.
> 초대코드 = 학원(테넌트) 지정용(= 아이 연결용 아님). 학원은 **학부모 명단 엑셀 업로드 → 초대코드 일괄 자동 발송**(별도 슬라이스 B, §9).
> 이하 본문의 "전화 대조 자동조회" 서술은 이 개정으로 대체됨.

## 2. 가져올 디자인 원칙 (ZEM → PACEFOLIO)

| 원칙 | ZEM 근거 | PACEFOLIO 적용 |
|---|---|---|
| **한 화면 한 행동** | 하단 풀폭 단일 다크 CTA, 보조는 아웃라인/텍스트 | 주 CTA = `--accent-strong` 풀폭 1개, 보조 = 텍스트 버튼 |
| **넓은 여백·짧은 문구** | 2줄 헤드라인 + 2~3줄 서브 | `PageHeader`: 큰 제목 2줄 + 서브 2~3줄, 상세는 접힘/별도 |
| **공통 화면 구조** | [진행표시][큰 제목][짧은 설명][콘텐츠][하단 고정 CTA] | `OnboardingLayout`로 전 화면 고정, 버튼 위치 불변 |
| **캐러셀 = 큰 일러스트 + 2줄/2줄 + dot** | 8슬라이드, 마지막만 CTA | 4+마지막 슬라이드, dot 인디케이터, 마지막만 `시작하기` |
| **권한 요청 전 설명** | 앱 설명 화면 → OS 팝업, 권한별 사유 | `PermissionGuide`: 알림 가치 설명 → (실서비스)OS 요청, 거절해도 이용 |
| **동의 필수/선택 분리** | 필수3/선택2 태그·전체동의·상세 | `AgreementList`: 필수/선택 태그, 전체동의, 마케팅=선택 |
| **부드러운 카드** | 큰 라운드 + 약한 그림자, 플로팅 | 기존 토큰 `--radius-lg(16)`·`--shadow-card`, 인풋 `--radius-md(12)` |
| **깔끔한 색** | 흰/옅은 배경 + 1 포인트, 빨강=오류만 | 기존 clean 토큰(teal 1액센트) 그대로. 경험영역은 부드러운 구분색 |

## 3. ⛔ 적용하지 않을 ZEM 기능 (아이 연결 없음)

ZEM 화면 15·16이 보여주는 **아이 연결 메커니즘 전체를 만들지 않는다.**

- ❌ 아이 검색 / 아이 이름·학원 검색
- ❌ 연결 신청(request) · 연결 코드 입력 · QR 연결
- ❌ 아이가 승인 / 다른 (법정)보호자 승인 대기 (ZEM 16의 2열 승인 카드)
- ❌ 별도 아이 연결 상태 관리 · "아이 연결하기" 버튼

**PACEFOLIO 흐름 (개정 2026-07-19):**
```
학원이 학부모 명단(엑셀) 업로드 → 초대코드 일괄 자동 발송   ← 슬라이스 B(§9, 원장/콘솔)
→ 보호자가 초대링크/코드로 PACEFOLIO 진입 (→ 어느 학원인지 결정)
→ 보호자 휴대폰 본인인증
→ 약관 동의
→ 보호자가 아이를 직접 등록 (이름·생년·프로그램, 형제는 "+아이 추가")
→ 보호자 홈
```
- **아이 검색·아이 연결코드·QR·아이 승인·타 보호자 승인 없음** — 부모가 자기 아이를 *입력*해서 만드는 것이라 ZEM 승인 핸드셰이크(화면 15→16)를 회피.
- 초대코드는 **학원(테넌트) 지정용**이며 코드에 이미 `guardianInvites`/`academyInviteCode` 인프라가 존재(§6).
- 기존 원생 데이터 매칭이 없으므로 "전화 대조 자동조회" 백엔드 재설계는 **불필요**(초기 요청서 대비 단순화).

## 4. PACEFOLIO 화면 변환 (와이어프레임 개요)

전 화면 `OnboardingLayout` = `[상단(진행 dot / 뒤로)] [큰 제목 2줄] [짧은 설명] [콘텐츠] [하단 고정 주요 CTA] [보조 텍스트버튼?]`.

| # | 화면 | 제목(요지) | 콘텐츠 | 주 CTA | 보조 |
|---|---|---|---|---|---|
| O1 | 온보딩 캐러셀 4+1 | 슬라이드별(아래 §4.1) | 성장 일러스트/카드 + 2줄 서브 | (마지막만)`시작하기` | 스와이프·dot·건너뛰기 |
| O2 | 초대코드/학원 확인 | "어느 학원에 등록하나요?" | `InviteCodeField` → `AcademyBadge` (코드 없으면 `AcademyPickRow` 목록) | `학원 확인`→`네, 시작할게요` | `학원 찾기` |
| O3 | 휴대폰 번호 입력 | "본인 명의 휴대폰 번호를 알려주세요" | `PhoneField`(통신사·번호) + 본인확인 안심 문구 | `인증번호 받기` | — |
| O4 | 인증번호 확인 | "문자로 받은 인증번호 6자리를 입력해요" | `OtpField`(6칸)·타이머·재전송 | `확인` | 번호 다시 입력 |
| O5 | 약관 동의 | "시작하기 전, 약관에 동의해 주세요" | `AgreementList`(필수/선택·전체동의) | `동의하고 계속하기` | 항목 상세 |
| O6 | **아이 등록(직접 입력)** | "우리 아이를 등록해요" | `ChildFormCard`(이름·생년·`ProgramPicker`) + `+아이 추가`(형제) | `등록 완료` | 아이 삭제 |
| O7 | 알림 권한 안내 | "아이의 새로운 성장을 놓치지 않도록" | `PermissionGuide`(수업기록·뱃지·코치이야기) | `성장 알림 받기` | `나중에` |
| → | 보호자 홈 | (기존 `/parent`) | — | — | — |

> 진입: 초대링크 `?code=WG2025` 로 오면 O2에서 학원 자동 확인. 데모 코드 = `WG2025`(원더짐)·`GD2025`(강동).

### 4.1 온보딩 캐러셀 카피(요청서 §5 준수)
1. **오늘의 수업 경험** — "오늘 어떤 경험을 했는지 한눈에 확인하세요"
2. **움직임 경험지도** — "참여할수록 움직임 경험이 쌓여요"(균형·이동·조작)
3. **기술 클리어와 뱃지** — "하나씩 완성하는 아이만의 성장 여정"
4. **성장보고서** — "비교가 아닌 아이 자신의 성장을 기록해요"
5. **마지막** — "아이의 모든 성장을 PACEFOLIO에서 만나보세요" → `시작하기`

> 점수·순위·비교·"잘함/못함" 금지. 브레인스포츠 = 경험 횟수·다양성·꾸준함 / 액티브스포츠 = 개인 기술 진도·코치 확인 뱃지. 실제 활동명·기술명 하드코딩 금지(중립 샘플).

### 4.2 보호자 홈 정보 순서 (요청서 §7 — 이번 범위 밖, 참고용)
`인사·프로필 / 자녀 전환 / 새 수업 기록 / 이번 달 움직임 경험 / 최근 뱃지 / 코치의 한마디 / 가정 연계 / 학원 공지`. 기능 메뉴 나열 금지, 성장 먼저. (현재 `/parent` 홈은 이미 실연결 상태 — 이번 작업은 온보딩·가입 신규 흐름만, 홈은 진입점만 연결.)

## 5. 개인정보/보안 점검

- ZEM 이미지 내 실명·이메일·생년월일·전화·아이 이름·학교·주소 → **문서·코드·테스트 데이터 어디에도 미복사.** 테스트/목업은 중립 샘플명(예: 도담·서준) 사용.
- 본인인증·인증번호는 **이번 범위=시뮬레이션**(실 PASS/SMS 미연동, 유저 확정). 실 전화번호를 로그·테스트 화면에 노출하지 않는다.
- 금액은 개인정보(헌법) — 온보딩·홈 진입에서 금액 비노출 원칙 유지.

## 6. 현재 코드 구조 조사 결과 (실연결 관점)

- **디자인 토큰**: `apps/web/app/tokens.css` v1.1 (SSOT, 디자인 터미널 흡수). teal 1액센트, 라운드 8/12/16/pill, 그림자 3단계, 버튼/인풋 44px, 터치 최소 44px, WCAG AA 색. **이번 작업은 tokens.css를 수정하지 않고 기존 토큰만 사용**(디자인 터미널 영역 보호).
- **공용 프리미티브**: `components/ui/index.tsx`(Button·Card·Tag·ProgressBar·cn 등), `components/mobile/MobileShell.tsx`(`PhoneFrame`·`AppHeader`·`AppScroll`). 재사용.
- **보호자 앱**: `app/parent/*` — `layout.tsx`가 `ParentProvider`+`ParentNav`(하단탭)+`Sheets`+`Toast`. 온보딩은 **하단탭이 없어야** 하므로 `/parent` 밑에 두면 안 됨 → **신규 최상위 그룹 `app/onboarding/`**(자체 layout=PhoneFrame만).
- **인증/연결 백엔드(중요)**:
  - 인증은 **소셜 OAuth만**(kakao 앵커·naver·google·apple) + dev 로그인. **휴대폰/SMS-OTP 엔드포인트 없음.**
  - 보호자 전화 → 원생 **직접 조회 쿼리 없음.** 전화 대조는 `POST /academies/:id/guardian-links`(링크 생성) 내부 게이트에만 존재하고, **추가로 childName+childBirth + 선발급 OTP 세션을 요구**하며 쌍둥이(다중 일치)는 자동 선택 안 함(PENDING).
  - 연결 후 자녀 목록: `GET /academies/:id/my-children` → `api.myChildren(academyId)` (존재).
  - 웹→API: `packages/api-client`(`createApiClient({baseUrl:"/api"})`, CSRF 자동, zod 검증). `apps/web/proxy.ts`가 `/parent` 등 역할 가드(GUARDIAN).
- **결론(백엔드 실연결 — ✅ 구현됨 2026-07-19 슬라이스 A)**: 모델이 "전화 자동조회"에서 "**초대코드 진입 + 아이 직접 등록**"으로 바뀌어 전화 매칭 재설계는 불필요해짐. **실 서버·DB 연결 완료**:

## 9. 슬라이스 B — 학원 대량 초대 (다음 작업 · 원장/콘솔)

학부모가 초대코드로 진입하려면 **학원이 코드를 나눠줘야** 한다. 한 명씩 수동 발송은 느리므로 **명단 기반 일괄 발송**을 만든다.

**흐름(원장/데스크 · PC 콘솔):**
```
학부모 명단 엑셀/CSV 업로드 (이름·전화번호)
→ 미리보기·검증 (중복·형식) — 기존 CSV 가져오기 스테이징 패턴 재사용(programs/imports)
→ 커밋 → 학부모별 초대코드 생성 (guardianInvites)
→ 일괄 자동 발송 (문자/카카오 알림톡) — 실 연동 전엔 시뮬
→ 발송·가입 현황 추적 (미발송·발송·가입완료), 개별 재발송·수동 발송
```
- **토대**: `guardianInvites`/`guardianInviteRedemptions` 테이블 + PC의 CSV 스테이징(업로드→미리보기→커밋→되돌리기)이 이미 존재 → 재사용.
- **실 연동 필요**: 문자/알림톡 발송 채널(현재 미연동 — 시뮬). 개인정보(전화번호) 최소 보관·암호화(기존 `crypto-pii` 정합).
- **위치(예상)**: `app/pc/students` 또는 신규 `app/pc/invites`. 이번 슬라이스 A(보호자 온보딩)와 분리.
