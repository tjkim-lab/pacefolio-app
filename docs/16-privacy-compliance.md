# 16. 개인정보·보안 컴플라이언스 — CPO 축 정리 (2026-07-18)

> CISO 축(시스템 방어)은 개발 과정에서 상당히 채워짐 — 권한·테넌트 복합 FK·
> 감사 로그·세션 해시·CSRF·웹훅 fail-closed·부정 테스트(docs/11 + 리뷰 13차).
> 이 문서는 비어 있던 **CPO 축(적법한 수집·이용·파기)** 의 실행 문서다.
> ⚠️ 전부 **법률 검토 전 초안** — 확정은 변호사 검토 + TJ 승인 후.

## A. 처리자 ↔ 수탁자 구조 (최우선 결정 — 사업 구조에 영향)
SaaS 의 고전 쟁점: **학원(원장)이 개인정보처리자, PACEFOLIO 는 수탁자**인가?

| 구조 | 의미 | 우리에게 |
|---|---|---|
| **수탁자 모델(권장 초안)** | 학원이 처리자 — 수집 주체·동의 획득 책임은 학원. 우리는 처리 위탁받은 자 | 학원별 **표준 위·수탁 계약** 필요(가입 약관에 포함 가능). 우리 의무 = 안전조치·재위탁 제한·감독 수용. 유출 시 1차 책임 소재가 명확 |
| 독립 처리자 모델 | 우리가 직접 수집·이용 주체 | 동의 전 과정 우리 책임 — 부담 큼. 앱 직접가입(보호자) 부분은 사실상 이 성격 혼재 |
| 혼합(현실) | 원생 데이터=수탁 / 보호자 계정·마케팅 동의=우리가 처리자 | **가장 유력** — 처리방침에 두 지위를 구분 명시 |

**결정 필요(TJ+법률)**: 혼합 모델 채택 여부 + 표준 위·수탁 조항을 학원 이용약관에 내장할지.
근거 자료: 만 14세 미만 법정대리인 동의(제22조의2)는 우리 보호자-자녀 OTP 연결이
기술적 구현 — 계약서에 이 절차를 동의 획득 방법으로 명시하면 정합.

## B. 보유기간 · 파기 정책 초안 (수치는 제안 — TJ 확정)
| 데이터 | 제안 보유기간 | 근거·비고 |
|---|---|---|
| 계약·결제·환불 기록 | **5년** | 전자상거래법(대금결제·재화 공급 기록) — 법정 |
| 소비자 불만·분쟁 처리 | **3년** | 전자상거래법 — 법정 |
| 원생 프로필·출결·성장 기록 | 퇴원 후 **1년** 보관 → 파기(분리보관 후) | 재등록 편의 vs 최소화 균형 — 제안 |
| 건강 메모(민감정보) | 퇴원 후 **즉시 파기** (사고 기록은 분쟁 시효까지 분리보관) | 민감정보 최소 보유 원칙 |
| 수업 사진 | 게시 후 **1년** 자동 삭제(보호자 다운로드 기간 고지) | C5 보관기간과 동일 결정 — 스토리지 정책에 반영 |
| 채팅 기록 | 퇴원·탈퇴 후 **1년** (분쟁 대비) → 파기 | soft delete 원문 보존은 보유기간 내에서만 |
| 세션·접근 로그 | **3개월~1년** | 통신비밀보호법 3개월(로그인 기록) 이상 |
| 탈퇴 계정 | 즉시 파기, 법정 보존 항목만 **분리보관** | docs/09 계약과 정합 — 기간 수치를 위 표로 확정 |
구현 연결: 파기는 배치 worker(EXPIRED·삭제 트랙)로 — Outbox 소비 worker 와 같은 인프라.

## C. 침해사고 대응 절차 (1페이지 — 유출 인지 시 그대로 실행)
1. **즉시(1시간 내)**: 유출 경로 차단 — 해당 세션 전체 폐기(`logout-all`)·의심 계정 SUSPENDED·필요 시 API 키 회전
2. **당일**: 영향 범위 확정 — AuditLog·접근 로그로 누가·무엇을·언제(우리 감사 체계가 이걸 위해 존재)
3. **72시간 내**: 개인정보보호위원회 신고(1천 명 이상 또는 민감정보·아동 데이터면 규모 무관 신고 권장) + **정보주체(보호자) 통지** — 유출 항목·시점·대응·문의처
4. **학원(처리자) 통지**: 수탁자 지위에서는 처리자인 학원에도 즉시 통지 의무
5. **사후**: 원인 분석 → 재발 방지 코드 반영 → 감사 기록. 대응 기록 자체를 보존
연락 창구: CPO = TJ (tj.kim@wondergym.co.kr) — 처리방침에 공개.

## D. 저장 암호화(at-rest) 설계 — 구현 착수
현황: 카드번호 **미저장**(설계로 회피) · 주민번호 없음 · 세션 tokenHash-only.
**공백 = 전화번호가 DB 평문**(표시만 마스킹). 대상과 방식:

| 필드 | 방식 | 이유 |
|---|---|---|
| users.phone · registered_guardian_contacts.phone · verified_phone | **hash(HMAC-SHA256+pepper) 컬럼**(매칭용) + **enc(AES-256-GCM) 컬럼**(표시용) | OTP 결합·선등록 대조가 동등성 매칭이라 결정적 hash 필요, 원문 표시는 복호화로 |
| 건강 메모·사고 기록 본문 | AES-256-GCM (매칭 불필요 — enc only) | 민감정보 |
| 채팅 body 중 HEALTH category | 동일 | 〃 |
키 관리: `PACEFOLIO_PII_KEY`(32B)·`PACEFOLIO_PII_PEPPER` env — 프로덕션 미설정 시 부팅 실패(fail-closed),
dev 는 고정 키 + 경고. 키 회전은 keyId prefix(v1:) 로 설계.
**이 커밋에서 crypto-pii 모듈+테스트 구현 — 컬럼 전환(마이그레이션+링크 플로우 배선)은 다음 배치(Task 등재).**
