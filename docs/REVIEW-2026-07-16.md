## 0. 리뷰어 종합 판정

> **UI 프로토타입으로는 충분히 좋지만, 화면별 API를 바로 연결하기엔 데이터·상태·권한·결제의 기준점이 아직 부족하다. 화면을 더 추가하지 말고, 먼저 기반을 고정하라.**

가장 먼저 고정해야 할 **4대 기반**:
1. **Invoice ID 기준** 결제 상태 + **Payment Allocation**(합산결제→원생별 배분)
2. 원장·코치·학부모를 잇는 **공통 이벤트·엔티티**
3. **서버 권한·멀티테넌트 격리**
4. **사진 동의** 버전·범위·철회 모델

## 1. P0 — 백엔드 착수 전 반드시 해결

- **P0-1 공용 데이터 원천 통합.** 앱별 `_data.ts`가 각자 세계관 → 원생 수·이름·청구금액·수납기간이 앱마다 충돌. `/stage`도 각 앱 독립 mock이라 "결석접수→코치명단→실출결→원장할일→보호자알림" 흐름이 안 이어짐. → `lib/domain/`(types·enums·state-machines·calculations) + `lib/fixtures/`(단일 원본) 만들고, 앱별 `_data.ts`는 **공용→ViewModel 변환만**.
- **P0-2 이름이 아닌 ID 기준 상태.** `invStatus[childName]` 위험 → `invoiceStatus[invoiceId]`, `attendanceStatus[classSessionId+participantId]`. 필수 ID: organization/academy/user/participant/guardian/guardianParticipantLink/enrollment/program/class/classSession/billingPeriod/invoice/invoiceLine/payment/paymentAllocation/refund/supportTicket/notification. 화면엔 이름, 상태·API는 항상 ID.
- **P0-3 앱 간 이벤트 흐름 확정.** `ATTENDANCE_NOTICE_CREATED → OWNER_TASK_CREATED → COACH_ROSTER_UPDATED → ACTUAL_ATTENDANCE_RECORDED → OWNER_TASK_RESOLVED → GUARDIAN_NOTIFIED`. 이벤트 필드: eventId/eventType/academyId/participantId/classSessionId/actorId/actorRole/occurredAt/idempotencyKey/correlationId/causationId/payloadVersion. (단, "최종 기준 데이터가 어디 저장되나"부터 정할 것 — 이벤트가 데이터모델 대체 못 함.)
- **P0-4 수납기간 표기 통일.** "3분기/9~11월/12월시작" 혼재 제거 → 저장은 `billingPeriodId + periodStart(YYYY-MM-DD) + periodEnd + cycleMonths`, 화면은 `YYYY-MM-DD ~ YYYY-MM-DD`. "분기"는 회계분기 혼동 → 제거/보조설명만. 모든 숫자에 단위(93명·93건·6가족·23회…).
- **P0-5 결제·환불 = 서버 단일기준.** 회차·휴무차감·중간입회 일할·수강료·차량비·형제/다종목 할인·중복순서·세금·환불예상·수납률·PG상태 모두 서버 계산, 프런트는 표시+확인만. 멱등키(청구확정·대량알림·결제준비/승인·자동결제등록·재시도·환불요청/승인). 안전장치: PG토큰·카드원문 저장금지·webhook 서명검증·event ID 중복방지·서버 금액 재검증·중복결제 방지·부분결제/환불·타임아웃 후 PG 재조회·UI성공 ≠ PG최종승인·감사로그.
- **P0-6 인증·권한·테넌트 격리.** 모든 요청에서 로그인→역할→소속조직→접근가능 academyId→대상 리소스 academyId→필드·행동 권한 검증. 클라이언트가 준 role/academyId/ownerId/coachId/participantId/permission/supportViewEnabled **신뢰 금지**. 권한 테스트(타학원 participantId 직접요청, 타보호자 자녀조회, 퇴사코치 토큰재사용, 코치가 결제금액 조회, 원장이 Admin API, SupportView 만료 후 재요청, 마스킹 URL우회).
- **P0-7 사진 동의 모델.** `photoChecked:boolean` 불가 → assetId/participantId목록/guardianConsentId/consentPolicyVersion/consentedAt/expiresAt/revokedAt/allowedPurpose/allowedAudience/academyId/classSessionId. 목적·범위 분리(개별전달/반공유/내부기록/홍보/외부광고/SNS). 범위 확대·철회 시 발송시점 재검증.
- **P0-8 개인정보·티켓 연결.** CS 티켓 ↔ academyId/participantId/guardianId/classSessionId/invoiceId/paymentId/refundId/notificationId. + 보호자-원생 OTP·관계검증·법정대리인·동의버전·건강정보 최소노출·마스킹·해제사유·감사·보관기간·삭제/익명화·탈퇴/퇴원 처리.
- **P0-9 5개 앱 배포 경계 결정.** (A) 단일 Next + Route Group + 서버 미들웨어 + Admin API 분리, vs (B) 앱별 배포 분리(monorepo `apps/*` + `packages/*`). 최소한 **Admin은 별도 배포/명확한 서버 경계 권장**. → **유저 결정 필요.**
- **P0-접근성** 화면 확대 차단(`maximumScale:1`, `userScalable:false`) 제거. ✅ **2026-07-16 반영 완료.**

## 4. 역할별 UX 개선 요지
- **원장 모바일**: 홈 정보 중복(캐러셀↔업무목록) 축소, 순서 긴급→오늘업무→수납→코치전달→정원. 모바일=확인/승인/간단수정, 대량청구·회차계산은 PC로.

- **학부모**: 홈 우선순위 오늘할일→결제/마감→새소식. Q&A 검색화면 분리(검색·카테고리·긴급문의·문의전환·participant/invoice/session 자동연결). 홍보↔필수업무 분리.

## 8. 우리 팀 반영 트리아지 (← 우리 액션)

**① 헌법 정합성 — 완벽히 일치.** 리뷰의 결론("화면별 API 붙이지 말고 기반부터")은 우리 헌법 **"목업 확정 전 DB 착공 금지"**와 같은 말. P0 대부분은 **DB 착공이 아니라 "착공 전 스펙+프론트 mock 일관성"** 작업이다. → **DB는 여전히 안 만든다.** 지금 할 일 = 도메인 스펙 문서 + 공용 fixture 정비 + 접근성/구조 정리.

**② 개발 vs 디자인 분리(메모리 준수).** 접근성 중 **시각 항목(글자크기·터치영역·명암·warm/clean)**은 디자인 터미널 몫 → 인계. **코드·구조(viewport·ID·fixture·aria·focus-trap·컴포넌트 통합·상태머신·권한·OpenAPI)**는 이 개발 세션.

**③ 이번 턴에 반영 완료**
- 저장소 비공개 전환
- 접근성 P0: 화면 확대 차단 제거 (`app/layout.tsx`)
- 본 리뷰 + 트리아지 문서화
